# Phase 1：理解需求

## 1.1 捕获用户意图

用户可能的表达方式：
- "我需要一个在看板上展示燃尽图的功能"
- "帮我做一个按钮，点击后自动创建关联工作项"
- "我想在详情页加一个标签页显示测试报告"
- "做一个拦截器，提交时检查必填字段"

**不需要用户说"创建插件"**——AI 识别到需求需要自定义能力承载，自动进入插件开发流程。

## 1.2 确认功能理解

此阶段确认功能需求。**不主动推导点位类型**，大多数用户对插件点位没有概念。

### 用户已指定点位类型

如果用户在描述需求时**已经明确指定了点位类型**（如"我需要一个轻应用组件"、"加个看板点位"、"做一个拦截器"），则**直接采用用户指定的点位**，在确认模板中体现：

```
我理解你的需求是：

📋 功能：[一句话描述功能]
📌 预期效果：[描述用户期望看到的界面/交互效果]
🧩 点位类型：[用户指定的点位类型]

理解正确吗？如果需要调整可以告诉我。
```

**用户已明确指定的点位，Phase 2 中直接使用，不再由 AI 另行推荐替代方案。**

### 用户未指定点位类型

```
我理解你的需求是：

📋 功能：[一句话描述功能]
📌 预期效果：[描述用户期望看到的界面/交互效果]

理解正确吗？如果需要调整可以告诉我。
```

**用户确认功能理解后进入 Phase 2。** 点位类型将在 Phase 2 拉取 schema 后再分析推荐。

## 1.3 断点恢复

若用户中途回来继续，检查当前目录状态：
- 无 `plugin.config.json` → 从 Phase 2 开始
- 有 `plugin.config.json` 但无代码 → 检查点位是否已配置，从 Phase 2 或 3 开始
- 有代码 → 从 Phase 3 调试或 Phase 4 发布开始
